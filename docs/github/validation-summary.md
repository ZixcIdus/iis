# Сводка по валидации

Этот документ фиксирует не весь массив результатов, а короткую и честную картину того, как проект выглядит с точки зрения текущей проверки.

Главный принцип:

- строгая аргументация строится прежде всего на `strict`-сценариях;
- `proxy` и `hybrid` используются как дополнительная, а не основная линия;
- сильные результаты и слабые результаты описываются одинаково явно.

## Основные режимы проверки

- `strict` - используются только реально измеренные прямые признаки;
- `proxy` - разрешены proxy-замены и мягкие якоря;
- `hybrid` - используются все реально доступные прямые признаки, а недостающие части могут компенсироваться proxy-слоем.

## Главные датасеты для строгой линии

Основой доказательной части лучше считать:

- `ds002724 / strict`
- `ds002722 / strict`

Именно здесь у проекта есть наиболее честная проверка линии `V4 -> V6`.

## Краткая таблица ключевых сценариев

| Сценарий | Лучшая версия по текущему utility rank | Краткий вывод |
| --- | --- | --- |
| `ds002724 / strict` | `V6` | Лучший текущий строгий практический сценарий проекта |
| `ds002722 / strict` | `V6` | `V6` устойчивее как итоговое ядро, хотя отдельные метрики требуют осторожной интерпретации |
| `DREAMER / strict` | `V4` | Сценарий частично ограничен по доступности HRV-компонент и не должен быть главным аргументом |
| `WESAD / hybrid` | `V6` | Полезный дополнительный сценарий устойчивости |
| `CASE / hybrid` | `V6` | Полезный дополнительный сценарий динамики и внешней проверки |

## Что показывает ds002724 / strict

Это сильнейший строгий сценарий текущего проекта.

Сводно:

- `V6` занимает `utility rank = 1`;
- `effect_abs = 0.4335`;
- `distribution_overlap = 0.7992`;
- `reliability_level = high`.

Практический вывод:

- `V6` здесь выглядит как главное рабочее strict-ядро;
- `V4` остается важной базой, но уступает как итоговая архитектура;
- `V5` может быть сильным промежуточным шагом, но не должен заслонять `V6`.

## Что показывает ds002722 / strict

Здесь картина более тонкая.

Сводно:

- `V6` также занимает `utility rank = 1`;
- `effect_abs = 0.0302`;
- `distribution_overlap = 0.7895`;
- `relative_sensitivity = 0.1568`;
- `reliability_level = high`.

Важная оговорка:

- по отдельным метрикам `V4` не всегда выглядит слабее во всем;
- но как итоговая рабочая архитектура `V6` выглядит предпочтительнее;
- именно поэтому в публичной подаче лучше говорить не "V6 сильнее по каждой метрике", а "V6 лучше как обобщенное практическое ядро".

## Что делать с DREAMER

`DREAMER / strict` не стоит делать центральным доказательством.

Причины:

- доступность HRV-признаков там ограничена;
- сценарий не эквивалентен главным strict-наборам;
- результаты полезны как дополнительный тест, но не как основной фундамент вывода.

Именно поэтому `DREAMER` лучше использовать аккуратно:

- как внешний контроль;
- как индикатор чувствительности модели;
- как повод не делать слишком сильных универсальных заявлений.

## Где находятся исторические версии V1-V3

Версии `V1-V3` могут показывать сильное разделение в `proxy`-режимах, но это не должно использоваться как главный аргумент зрелости модели.

Корректная формулировка:

- это важные исторические этапы;
- они показали потенциал идеи;
- но их методическая чистота ниже, чем у линии `V4 -> V6`.

## Итог по валидации

Если сжать весь проект до нескольких аккуратных утверждений, то они будут такими:

1. Линия `V4 -> V6` действительно лучше подходит для честной конкурсной подачи, чем ранние proxy-версии.
2. `V4` важен как первая негормональная база, но не как финальный лучший результат.
3. `V6` сейчас является главным практическим strict-ядром.
4. `V7` интересен как инновационная ветка, но пока не должен подаваться как универсально подтвержденный итог.
